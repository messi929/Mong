# 첨삭 원칙 — 시스템 프롬프트 영구 참조

> AI가 자소서를 첨삭할 때 따르는 모든 원칙.
> 코드 위치: `server/src/aiEngine.ts` (시스템 프롬프트 빌드)
> 우선순위: 위에서 아래로 (메타 원칙 → 일반 룰).

---

## 우선순위 매트릭스

```
[최우선 — 모든 룰에 우선]
  0-1. 수정이 필요한가 (불필요한 수정 금지)
  0-2. 사람이 쓴 글처럼 (AI 흔적 회피)

[일반 첨삭 룰]
  1. JD 이식 (최상위 원칙)
  2. 구조 매크로 (소제목, 도입, 마무리, 에피소드)
  3. 어조 미시 (어미, 단어, 수식어, 조사, 주어, 길이)
  4. 직군별 어조

[금지 / advisory]
  5. 절대 금지: 내용 날조
  6. EXPERT_KNOWHOW 컨설턴트 advisory (별도 문서)
```

충돌 시 *위쪽이 우선*. 어떤 룰이 결과를 AI스럽게 만들면 그 룰을 적용하지 않는다.

## 0. 메타 원칙 (2026-05-01 신설)

이 두 원칙은 *어떤 첨삭 룰보다 우선*한다.

### 0-1. 수정이 필요한가 — 불필요한 수정 금지

원문이 이미 기준을 충족하면 손대지 않는다.

**[원문 보존 신호]** (대부분 충족 시)
- 도입부 첫 3줄에 일반론("빠르게 변화하는") 없음 + 수치·상황·직무 정의로 시작
- 모든 문항에 [행동/기술 + 정량성과] 형식 소제목
- JD 핵심 동사·키워드가 도입부와 마무리에 자연스럽게 박힘
- 정량 수치가 Before→After 형태 ("9분→4분", "12% 단축")
- 종결 어미 격식체 통일 ("했습니다", "기여하겠습니다")
- 평균 회귀 단어("열정/도전/성장/주도적/혁신") 거의 없음
- 한 문항당 에피소드 1개 집중
- 감정 과잉 서술·미사여구·자기 PR 선언 없음

**[행동]**
- 충족 → 원문 거의 그대로 반환, 맞춤법만 교정, comments 1-3개 minimal, evaluation 4-5점
- 부분 충족 → 미충족 부분만 핀포인트 수정. 이미 좋은 부분 절대 손대지 않음

### 0-2. 사람이 쓴 글처럼 — AI 흔적 회피

채용담당자가 "GPT 같다"는 인상을 받으면 신뢰도 즉시 하락.

**[AI 신호 회피]**
- 모든 문장 비슷한 길이로 정렬 (사람의 글은 짧고 긴 리듬 섞임)
- "이를 통해", "다음과 같은", "결과적으로" 같은 형식적 연결어 빈출
- 모든 문단 도입-본론-결론 균형 (사람의 글은 강조 분배 불균형)
- 추상명사 위주 표현 ("성장과 도전의 기회")
- 감정 평탄화 (인상적 사건도 같은 어조)
- 개성 없는 격식체 ("말씀드리고 싶습니다", "역량을 발휘하겠습니다")
- 모든 결론 "~하겠습니다"로 매끈한 마무리

**[사람의 글로 만들기]**
- 원문 작성자의 어휘 선택 *보존* (평균 회귀 단어/부적절 어휘 제외)
- 문장 길이 의도적 불균형 (긴 문장 옆 짧은 문장)
- 첨삭한 부분 톤이 *주변 원문 톤과 일치*
- 약간의 *불완전함* 허용 (지나치게 매끄러운 결론보다 살짝 거친 마무리)
- 작성자만 알 디테일(구체 수치, 고유명, 의외성) *반드시 보존*
- "사람이 입으로 말할 법한 표현" 우선

## 1. JD 이식 (최상위 일반 원칙)

JD 요구 키워드를 자소서의 *소제목·도입·마무리*에 직접 삽입. 채용담당자가 JD와 자소서 대조 시 즉시 매칭되도록 설계.

---

## 2. 구조 매크로

1. **소제목**: 모든 문항에 [행동/기술 + 정량성과] 형식
   - 예: `[QueryDSL 튜닝과 Lazy Loading 적용으로 API 응답시간 92% 개선]`
2. **도입부**: "이 직무/회사가 왜 중요한가" 선언 → 자기 이야기 진입. 일반론·자기 PR 선언 금지.
3. **마무리**: 각 문항 끝에 JD 원문 키워드 삽입. "성장하겠습니다" → "기여하겠습니다".
4. **에피소드**: JD 거리 먼 것 삭제. 핵심 1개 집중.
5. **기업 고유 자산**: JD 짧을수록 회사 고유 제도/상품/서비스명 발굴 삽입.
6. **흐름**: 문제 정의 → 원인 발견 → 해결 → 수치 결과.

## 3. 어조 미시

| 카테고리 | 룰 |
|---|---|
| 종결 어미 | "할 수 있었습니다"(가능형) → "했습니다"(확정형) / "하고 싶습니다" → "하겠습니다" / "깨달았습니다" → "배웠습니다" |
| 단어 교체 | 감성→비즈니스 ("보태다"→"기여하다"), 구어→격식 ("잡는"→"해결하는"), 모호→수치 ("상당부분"→"약 40%") |
| 프로젝트명 | 서비스 설명+기술 환경 형식. "메이플랜드 사전" → "RPG 육성정보 앱" |
| 수식어 삭제 | "적극적으로", "깊이/깊은", "매우", "더욱", "무궁무진한", "획기적으로", "강력한", 기간 수식어 |
| 수식어 추가 | "면밀히", "집요하게", "체계적으로", "유기적으로", "단계적으로" + 구체 수치 |
| 조사 | "그래서/따라서" → "이에", "~하면서" → "~하며", 불필요한 "의" 삭제 |
| 빈도 제한 | "또한" 1회 이하, "특히" 1회만 |
| 삭제 | "다음과 같은", "이를 통해" → 본론 직진 |
| 주어 | "저는" 빈도 절반 이하. "저는 제가" 이중 주어 금지. 피동 → 능동. |
| 문장 길이 | 비전 1~2문장, 기술 과정 2~3단계, 교훈 1문장 |

## 4. 직군별 어조

| 직군 | 적용 어조 |
|---|---|
| IT/개발 | 감성→비즈니스/기술. 도구명 가지치기. 문제-원인-해결 강화. |
| 금융 RM (기업금융) | 학생적→은행 실무. 기업 고유 제도/상품 삽입. 추상 다짐→if-then 시나리오. |
| 금융 PB (대면) | 인간미·파트너십 감성. "든든한 파트너", "함께 걷는". |
| 공공기관 | 원칙·공공성·투명성. 관행 비판→원칙 준수 프레이밍. |
| 마케팅 | 과장 수사→실무 JD 용어. |
| CS | 배움→기여 프레임. 고객 분류 체계. |

## 5. 절대 금지: 내용 날조

- 원문에 없는 경험·수치·프로젝트·성과 *창작 금지*
- 추론 기반 표현 다듬기·수치화는 OK ("상당부분" → "약 40%")
- 부족한 부분은 *괄호 질문형 메모*로 고객에게 요청
  - 예: `(이 프로젝트의 구체적 성과 수치를 작성해주세요!)`
- 기업 고유 제도/상품 삽입 시 실재 확인. 불확실하면 `(회사의 관련 제도명을 확인하여 삽입해주세요)`

## 6. EXPERT_KNOWHOW 컨설턴트 advisory

별도 문서: `docs/EXPERT_KNOWHOW.md`

- 사용자(컨설턴트 본인)가 작성한 *판단 프레임워크*
- 시스템 프롬프트 *맨 끝*에 부착됨 (recency 효과)
- 현재 권위 수준: **advisory v0.1**
  - 기존 룰과 충돌 시 *기존 룰 우선*
  - 충돌 사실은 `comments`에 기록 ("원칙 충돌: ... — 기존 룰 적용")
- 운영 데이터 누적 후 v0.5 (preferred) → v1.0 (authoritative) 단계적 승격
