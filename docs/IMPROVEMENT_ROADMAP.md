# 시스템 개선 로드맵 — 자체 검증 후 정리

> 직전 대화에서 제안한 개선안을 자체 비판적으로 재검토한 결과.
> 검증 후 *유지 / 조정 / 다운그레이드*된 항목을 분리해 표시.
> 시스템 전반은 `docs/CONSULTING_MECHANISM.md`, 노하우는 `docs/EXPERT_KNOWHOW.md` 참조.

---

## 0. 자체 검증 방법론

직전 제안에 5가지 질문을 적용해 재검토:

1. 실제 *증거*가 있었나, 아니면 추측이었나?
2. 구현 비용을 *과소평가*하지 않았나?
3. 임팩트를 *과대평가*하지 않았나?
4. *더 단순한 대안*을 너무 빨리 기각했나?
5. **1인 운영 + 18쌍 학습 데이터** 규모에 맞는가? (오버엔지니어링 점검)

특히 5번이 중요. 큰 시스템 패턴(자동 머지, 신뢰도 점수, 주기적 재계산)을 작은 운영에 박으면 운영 부담만 늘고 효과는 작다.

## 1. 검증 결과 요약

| 제안 | 직전 분류 | 검증 후 | 핵심 조정 |
|---|---|---|---|
| 컨설턴트 수정률 KPI | Phase 0 (필수) | **유지** | 단일 지표 → 주 지표 + 보조 지표 세트 |
| 학습 루프 폐쇄 (자동 머지) | Phase 1 | **다운그레이드** | 자동 머지 → 수동 검토 큐 (v1), 자동화는 v2 |
| 사례 → annotated diff 전환 | Phase 2 | **조정** | 즉시 전환 → 하이브리드(1 full + 2 diffs) |
| 노하우 최상위 권위 레이어 | Phase 3 | **조정** | 즉시 authoritative → advisory 단계 거침 |
| 다문항 3-pass | Phase 4 | **조건부 보류** | KPI 데이터로 실제 실패 확인 후 결정 |
| 모델 분리 / extended thinking | Phase 5 | **유지** | KPI 다음 |
| 프롬프트 캐싱 | Phase 5 (묶음) | **앞으로 당김** | 비용 절감 목적이면 Phase 0 병행 가능 |
| 객관 평가 지표 도입 | 위험 3 해결책 | **조정** | 주 지표 X, 보조 health check로 위치 변경 |
| confirmed 케이스 학습 등록 | 위험 1 해결책 | **조정** | 자동 등록 X, 명시 토글 (lazy 컨설턴트 가드) |
| 충돌 comments 기록 | 위험 2 해결책 | **유지+보강** | review UI 동시 설계 (없으면 write-only) |

## 2. 제안별 재평가

### 2-1. 컨설턴트 수정률 KPI ✅ 유지 (단, 보강)

**직전 제안:** `diff_ratio(first, final)`을 단일 메인 지표로.

**자체 검증 — 약한 부분:**
- diff_ratio는 *substantive 변경*과 *stylistic 변경*을 구분 못 함. 컨설턴트가 의미는 안 바꾸고 어조만 다듬은 케이스도 같은 점수.
- 단일 컨설턴트라 변동성은 낮지만, 컨설턴트 mood/시간압박이 점수에 들어옴.
- 목표값(target) 없이 추세만 본다면 "내려가면 좋은 거"인지도 불확실 (너무 낮으면 컨설턴트가 검토를 안 한다는 신호일 수도).

**조정안 — 보조 지표 추가:**

| 지표 | 종류 | 의미 |
|---|---|---|
| `edit_ratio` | 주 지표 | first→final diff (토큰 단위) |
| `edit_binary` | 주 지표 | 컨설턴트가 *손댔는가* (yes/no) — 가장 robust |
| `consultant_time_spent` | 보조 | 검토 소요 시간 (UI에서 자동 측정) |
| `re_consultation_count` | 보조 | "AI 재첨삭" 버튼 횟수 |

**왜 `edit_binary`가 robust한가:** 컨설턴트가 *진짜로* AI 결과에 만족했는지의 가장 깨끗한 신호. ratio는 노이즈 많지만 binary는 단순. 두 지표 같이 보면 신뢰도 올라감.

**구현:** 30-50줄. LLM 호출 없음. 추가 latency 0. 기존 revision 데이터에서 자동 계산.

### 2-2. 학습 루프 폐쇄 ⚠️ 다운그레이드

**직전 제안:** `analyzeTrainingPattern` 출력을 구조화 JSON delta로 → confidence score로 자동 머지 → 매주 재계산.

**자체 검증 — 약한 부분:**
- N=18-50에서 confidence score는 통계적으로 의미가 약함. "3/3 = 100%"는 3건의 우연일 수 있음.
- 자동 머지 시스템 자체가 코드량/디버깅 비용 큼. 1인 운영에 *과한 인프라*.
- 더 단순한 대안: 분석 결과를 *검토 큐*에 쌓고 컨설턴트가 월 1회 일괄 검토. 수동이지만 안전하고 통제 가능.

**조정안 — 2단계로 분리:**

**v1 (이번 분기, 단순):** Manual Review Queue
```
analyzeTrainingPattern 결과 → 'pattern_review_queue' 테이블에 저장
admin 화면에 "검토 대기 N건" 배지
컨설턴트가 월 1회 일괄 검토:
  - 룰 채택 → style_profile에 수동 추가
  - 룰 기각 → 'rejected' 표시 (다음에 같은 패턴 또 나오면 무시)
  - 보류 → 다음 검토까지 대기
```

**v2 (운영 데이터 N=50+ 시):** confidence-scored 자동 머지
- v1에서 컨설턴트 채택/기각 이력이 쌓이면 그게 ground truth가 됨.
- 그 이력으로 자동 머지 임계값을 *경험적으로* 정함 (이론적으로가 아니라).
- 즉, v1이 v2의 학습 데이터를 만든다.

**왜 이렇게 바꾸나:** 자동화는 좋은데 *자동화의 정확도를 검증할 데이터*가 없으면 "자동화가 망쳐도 모름" 상태가 됨. 수동을 거쳐서 자동화 임계값의 근거를 만든 다음 자동화하는 게 안전.

### 2-3. 사례 → annotated diff 전환 ⚠️ 조정

**직전 제안:** 풀 텍스트 사례(draft/first/second/final 4단계) → annotated diff로 즉시 전환. 토큰 4-6K → 800-1.5K.

**자체 검증 — 약한 부분:**
- **토큰 추정 재계산:** 한국어 700-1000자 자소서는 1500-2200 토큰. 4단계 × 1500-2200 = 6000-8800 토큰/케이스. 3건 = **18-26K 토큰**. 직전 추정보다 큼.
- diff 포맷 효과는 진짜 큰데, **annotated diff가 학습 신호를 정말 보존하는지는 미검증 추측.** 풀 컨텍스트에서 오는 *누적 효과* (소제목 형식 + 도입부 변경 + 본문 압축이 함께 만드는 톤)가 있다.
- 즉시 전환했다가 품질 떨어지면 롤백 비용도 크다.

**조정안 — 하이브리드로 시작:**

```
3건 사례 = 풀 케이스 1건 (가장 최근 또는 가장 유사한 직군 1건)
        + annotated diff 2건 (나머지)
```

토큰: 6K + 2 × 1.5K = **약 9K** (현재 18-26K → 절반 이하).
효과 검증: KPI(`edit_ratio`)로 비교. 풀 3건 vs 하이브리드 3건 vs diff 3건 A/B.

검증 결과 따라 결정:
- 차이 작음 → diff 3건으로 더 다이어트
- 차이 큼 → 풀 케이스 비중 유지 또는 늘림

**annotated diff 포맷 (검증된 형태):**
```
[BEFORE] 적극적으로 협업하여 업무를 추진했습니다
[AFTER]  주도적으로 협업해 12% 일정을 단축했습니다
[CATEGORY] 평균 회귀 단어 교체 + 정량화
[CONTEXT] IT 기획자, 신입
```

CATEGORY는 EXPERT_KNOWHOW 또는 style_profile의 룰 ID와 매칭. CONTEXT는 직군·연차로 모델이 *언제 적용할지* 학습.

### 2-4. 노하우 최상위 권위 레이어 ⚠️ 조정

**직전 제안:** EXPERT_KNOWHOW.md를 시스템 프롬프트 *마지막*에 *최상위 권위*로 박음.

**자체 검증 — 약한 부분:**
- 방금 만든 v0.1은 **AI 생성 베이스라인**. 컨설턴트 검수 안 된 상태에서 최상위 권위로 박으면, 잘못된 지침이 다른 모든 룰을 덮어버림.
- "충돌 시 노하우 우선" 원칙이 잘못된 노하우를 *더 크게 잘못 적용*하게 만듬.

**조정안 — 권위 단계화:**

| 단계 | 권위 수준 | 시스템 프롬프트 표현 | 조건 |
|---|---|---|---|
| **v0.1 (advisory)** | 낮음 | "다음은 컨설턴트의 일반 원칙이다. 충돌 시 *기존 룰 우선*" | 검수 직후 |
| **v0.5 (preferred)** | 중간 | "다음 원칙은 신뢰도가 높다. 충돌 시 *원칙 우선이지만 comments에 충돌 명시*" | 5-10건 운영 후 |
| **v1.0 (authoritative)** | 높음 | "다음 원칙이 최우선. 충돌 시 무조건 따름" | 컨설턴트가 v0.5에서 80% 이상 결과에 동의 후 |

**왜 단계화하나:** 시스템 프롬프트의 권위는 *되돌리기 어렵다*. v0.1이 잘못 작동해도 KPI에서 즉시 안 보임 (다른 룰들이 덮어줌). v1.0에서 잘못 작동하면 모든 첨삭이 동시에 망가짐.

advisory 단계에선 노하우 충돌이 comments에 기록되고, 컨설턴트가 그 충돌을 보면서 노하우 자체를 다듬음. 즉, **노하우 v0.1이 운영 중에 v1.0으로 진화**하는 구조.

### 2-5. 다문항 3-pass ⏸️ 조건부 보류

**직전 제안:** Strategy pass → Per-question pass × N → Coherence pass. 단일 호출로는 다문항 안정 처리 불가.

**자체 검증 — 약한 부분:**
- "단일 호출로 다문항 잘 안 됨"은 *원리적 추측*. CONSULTING_MECHANISM Section 11 #7도 "분할 처리 필요한지 미정"이라고 명시.
- 3-pass는 latency 1.5-2x + 코드 복잡도 큼.
- **단일 호출이 70% 케이스에서 충분하면 비용 대비 가치 작음.**

**조정안 — KPI 분할 측정 후 결정:**

Phase 0 KPI에 segment 추가:
```
edit_ratio_by_question_count:
  - single_question: 평균 0.18 (예시)
  - 2-3_questions:   평균 0.25
  - 4+_questions:    평균 0.45 ← 여기서 큰 차이 나면 3-pass 정당화
```

만약 다문항에서 edit_ratio가 단문항 대비 2배 이상이면 → 3-pass 진행.
1.3배 이내면 → 단일 호출 유지하고 다른 곳(글자수 검증, 캐치프레이즈 일관성)만 후처리로 보강.

**임시 보강책 (3-pass 전):** 다문항 입력 시 user prompt 최상단에 명시:
```
[다문항 안내]
이 자소서는 N개 문항이다. 각 문항의 글자수 한도는 [800/1500/500]자.
모든 문항을 관통하는 한 줄 메시지 먼저 정하고, 각 문항이 그 메시지로 수렴하게 첨삭하라.
```

이것만으로도 단일 호출 품질이 꽤 올라갈 가능성. 측정해본 후 더 갈지 결정.

### 2-6. 모델 분리 / extended thinking / 캐싱 — 분리 결정

**모델 분리 (Opus 첨삭 / Sonnet 평가 / Haiku 패턴):**
- 직전 권장 유지. **단, KPI 도입 후.** 그래야 모델 변경의 효과를 측정 가능.
- 우선순위: 마지막.

**Extended thinking:**
- 직전엔 "strategy pass에서 빛난다"고 했는데 이건 **추측**. 단일 첨삭에서 효과 있을 수도 있음.
- 조정: Phase 0 KPI 운영되면 즉시 A/B 가능. Opus thinking on/off 비교. 결과로 결정.

**프롬프트 캐싱:**
- 직전엔 Phase 5에 묶었는데, 분리해야 함. **캐싱은 품질 문제를 안 푸는 비용 절감 도구**. 비용 부담이 있으면 *Phase 0와 병행 가능*.
- 단, 캐싱 도입 자체가 시스템 프롬프트 구조를 안정화시켜야 효과 큼 (캐시 invalidation 자주 일어나면 무의미). 노하우 v0.1 → v1.0 진화 중에는 시스템 프롬프트가 자주 바뀌니 캐싱 효과 떨어짐. **노하우 안정화 후 도입이 효율적.**

### 2-7. 객관 평가 지표 ⚠️ 조정 (주 지표 → 보조)

**직전 제안:** 5축 주관 루브릭의 절반을 객관 지표로 교체 (JD 키워드 매칭률, 글자수 활용률, 정량 표현 밀도, 금지 단어 빈도).

**자체 검증 — 약한 부분:**
- **JD 키워드 매칭률은 gameable.** 키워드 스터핑이 점수 올리지만 자소서는 망함. 좋은 첨삭일수록 JD 언어를 *자연스럽게* 녹이는 거지 매칭률 극대화가 아님.
- **금지 단어 카운트는 over-trigger.** "성장"이 정당한 맥락에서 등장할 수 있는데 무조건 깎으면 잘못된 신호.
- 이 지표들은 *quality 측정*이 아니라 *health check*에 가까움.

**조정안 — 위치 변경:**

| 지표 | 위치 | 용도 |
|---|---|---|
| `edit_ratio`, `edit_binary` | 주 지표 (KPI) | 시스템 품질의 ground truth |
| 글자수 준수율 | 보조 (health check) | 100% 미만이면 alert (룰 위반 신호) |
| 금지 단어 빈도 | 보조 (health check) | 임계값 넘으면 컨설턴트 검토 alert |
| JD 키워드 매칭률 | 분석용 (정보) | 추세만 보고 게이밍 안 함 |
| 정량 표현 밀도 | 분석용 (정보) | 추세만 |

**평가 모델은 여전히 분리 권장**(Sonnet 4.6 또는 다른 패밀리). 단, 평가 점수 자체를 *주 KPI로 삼지 않음*. 주 KPI는 컨설턴트 행동(`edit_*`).

### 2-8. 학습 토글 디폴트 (샘플링 편향) ⚠️ 조정

**직전 제안:** `confirmed` 케이스 (AI 결과 그대로 통과)도 학습 데이터로 자동 등록 (소량 샘플링).

**자체 검증 — 약한 부분:**
- "AI 결과 그대로 통과 = AI가 잘함"은 가설. **컨설턴트가 게을렀거나 시간 압박 상태**일 수도. 실제로 검토 안 한 케이스가 학습 데이터에 들어가면 노이즈.

**조정안 — 자동 등록 대신 *명시 토글*:**

```
마무리 모달:
  [ ] 이 사례를 학습 데이터로 추가 (수정 있을 때 기본 ON)
  [ ] 이 사례는 AI가 잘 처리했음 — 양성 사례로 추가 (NEW, 항상 OFF)
```

두 번째 토글은 컨설턴트가 *의식적으로* "AI가 진짜 잘했다, 이건 보존할 가치 있다"고 판단할 때만 켬. 게으른 통과와 의식적 인정을 분리.

운영 데이터: edited 케이스 vs confirmed 케이스 비율을 KPI에 표시. 너무 한쪽으로 쏠리면 알 수 있음.

### 2-9. 충돌 surfacing ✅ 유지 + 보강

**직전 제안:** 노하우 vs style_profile 충돌을 `comments`에 기록.

**자체 검증 — 약한 부분:**
- comments 필드는 이미 있음. 거기에 conflict 기록해도 *누가 읽나*. 컨설턴트는 자소서를 빨리 마무리하고 싶을 뿐, conflict 분석할 시간 없음.
- 결국 **write-only 데이터**가 됨.

**조정안 — review UI 동시 설계:**

```
admin 화면 / "충돌 검토" 섹션:
  최근 30일 발생한 conflict 모음
  - 노하우 X vs style_profile Y, 빈도 N건
  - 컨설턴트 결정: [노하우 우선 ✓] [style_profile 우선] [둘 다 잘못, 새 룰]
```

노하우 v0.5 → v1.0 승격 결정도 이 화면에서 데이터 기반으로 함. "노하우 우선이 맞았던 비율 X%"가 보이면 승격 가능 여부 판단 가능.

## 3. 새로 보인 이슈 (자체 검증 중 발견)

### 3-1. 시스템 프롬프트 안정성과 캐싱·학습 루프의 상호작용

직전엔 캐싱과 학습 루프를 따로 봤는데, 둘은 충돌 관계임:
- 학습 루프 활성 = 시스템 프롬프트 자주 변함 = 캐시 invalidation 잦음 = 캐싱 효과 ↓
- 캐싱 효과 극대화 = 시스템 프롬프트 안정 = 학습 루프 결과 반영 늦어짐

**해법:** 시스템 프롬프트를 *안정 부분*과 *변경 부분*으로 분리:
- 안정: 하드코드 룰, 노하우 v1.0 (캐시 가능)
- 변경: style_profile 최신 룰, 사례 (캐시 외부)

이걸 의식하면 두 시스템이 서로 발 안 밟음.

### 3-2. 평가가 평가 자체의 KPI를 못 가짐

평가 모듈을 모델 분리하고 객관 지표로 보강해도, **평가가 잘 작동하는지 자체를 어떻게 알 것인가**의 답이 시스템에 없음.

- 평가 = "이 자소서는 4점이다"
- 진짜 검증 = "이 자소서로 실제 합격/불합격 결과는?"

진짜 검증은 채용 결과 추적이 필요. 사용자가 클라이언트로부터 합격 결과를 receive하는 시스템이 있는가? 없으면 평가 신뢰도는 영원히 검증 안 됨.

**제안:** consulting 데이터 모델에 `outcome` 필드 추가:
```
outcome: 'pending' | 'document_passed' | 'document_failed' | 'final_passed' | 'final_failed' | 'unknown'
outcome_received_at: datetime
```

마무리 후 1-2개월 뒤 재방문 트리거 또는 클라이언트 자가 보고 UI. **이게 시스템의 진짜 KPI**가 되며, 모든 다른 지표(edit_rate, evaluation score 등)의 정합성을 사후 검증함.

장기적으로 가장 가치 있는 데이터.

### 3-3. 컨설턴트의 *시간*도 KPI

직전 제안에 빠졌던 것: 시스템의 목표는 단순히 "좋은 자소서 만드는 것"이 아니라 **"컨설턴트가 적은 시간으로 좋은 자소서 만들게 하는 것"**.

KPI에 추가:
- `consultant_active_time_per_consulting`: 첨삭 시작~마무리까지 컨설턴트가 *실제로 화면을 보고 있던* 시간 (idle 제외)
- 시간 단축이 *품질 유지*되며 일어나야 의미 있음. edit_rate와 같이 봐야 함.

## 4. 수정된 단계별 계획

- [ ] 충돌 review UI (Section 2-9)
- [ ] `confirmed_good` 명시 토글 추가 (Section 2-8)

### Phase 2: 노하우 도입 (advisory) (Phase 1과 병행)
**목표:** EXPERT_KNOWHOW v0.1 도입, 단 보수적으로.

- [ ] 컨설턴트가 v0.1 검수 → v0.2

- [ ] 시스템 프롬프트를 안정/변경 부분 분리 (캐싱 준비)

**목표:** 데이터가 필요하다고 말하면 진행.

## 5. 컨설턴트가 결정해야 할 열린 질문

자체 검증 후 답이 나오지 않은 것들 — 컨설턴트만 결정 가능:

1. **`outcome` 추적 — 클라이언트한테 합격 결과 물어볼 채널 있나?** 없으면 가장 큰 KPI를 못 만듦. 어떻게 만들 수 있나?
2. **노하우 v0.1을 검수할 시간 — 한 번에 몰아서 vs 사례 처리하면서 incremental?** Phase 2 진행 속도 결정.
3. **수동 검토 큐 — 월 1회 검토 페이스 가능한가, 아니면 분기 1회?** Phase 1 정의에 영향.
4. **admin 대시보드 우선순위 — 어떤 화면 먼저 만드나?** KPI 추세 / 충돌 검토 / 학습 큐 / 사후 결과 — 셋 다는 한 번에 못 함.
5. **다문항 자소서 비중 — 실제 의뢰의 몇 % 가 4문항 이상?** Phase 4 우선순위에 직결.
6. **A/B 운영 가능한가 — 같은 케이스를 두 시스템으로 돌려볼 시간/의지 있나?** 없으면 KPI 추세만으로 판단해야 함 (정확도 낮아짐).

## 6. 직전 제안 대비 가장 중요한 변화 (요약)

**직전:** 5단계 phased plan, 각 단계가 *진행 가정*.
**검증 후:** **Phase 0(측정)가 모든 후속 단계의 게이트.** 다른 단계는 KPI 데이터에 따라 *조건부* 진행.

직전엔 "Phase 0 → 1 → 2 → 3 → 4 → 5" 직선 진행을 가정했는데, 자체 검증 결과:

- Phase 1, 2는 Phase 0 직후 *병행* 가능 (서로 독립).
- Phase 3은 Phase 2 결과 안정화 후.
- Phase 4는 *조건부 보류*. 데이터가 필요성 입증 못하면 안 함.
- Phase 5는 가장 마지막. 단 캐싱은 Phase 3 후 도입 가능.

또 하나의 큰 변화: **노하우의 권위 부여를 단계화**. 직전엔 즉시 최상위로 박는 걸 가정했는데, 검수 안 된 노하우가 시스템을 망칠 수 있음. advisory → preferred → authoritative 3단계.

그리고: **`outcome` 추적이 없으면 시스템 전체의 진짜 KPI가 검증 불가**. 직전 제안에 이게 없었던 게 가장 큰 누락. 우선 채널 만들기부터 고민 필요.
