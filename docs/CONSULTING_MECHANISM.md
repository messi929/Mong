# Mong Consulting — 컨설팅 메커니즘 의논 문서

> 다른 LLM/사람과 첨삭 메커니즘을 깊게 의논하기 위한 자급자족 문서.
> 시스템 전반 레퍼런스는 `docs/PROJECT_GUIDE.md` 참조.

---

## 1. 프로그램 컨셉

세계적 수준의 자기소개서 컨설턴트(전재현)의 첨삭 패턴을 학습한 AI가, 컨설턴트 본인 또는 그가 위임한 사용자가 *동일하거나 더 높은 품질로* 첨삭을 수행하도록 보조하는 데스크톱 앱.

핵심 차별점:
- 단순 첨삭 도구가 아니라 **컨설턴트 본인의 사고와 판단을 복제하려는 시스템**
- 학습 데이터(18쌍 + 합격자소서 27건)를 명시적 규칙으로 추출하여 **스타일 프로파일** 형태로 누적
- 첨삭할 때마다 학습 사례가 다시 in-context로 주입되어 스타일 일관성 유지
- **명시적 학습 게이트** — 컨설턴트가 직접 손본 결과만 학습 데이터로 등록 (AI→AI 순환 차단)

---

## 2. 개발 형태 (요약)

```
[Electron 데스크톱 앱]  ←HTTP+JWT→  [Hetzner 서버]
React 19 + TS                     Express + SQLite
                                  Claude API (Sonnet 4)
```

- 데스크톱: 로컬 파일 파싱(Word/PDF/HWP), UI, 결과 내보내기
- 서버: 인증, CRUD, AI 호출, 학습 데이터 관리, 스타일 프로파일

---

## 3. 데이터 모델 (핵심)

```
clients (고객)
  └─ consultings (회사·직무 단위 첨삭 건)
        └─ revisions (단계별 본문)
              stage: draft(원본) | first(AI) | second(컨설턴트수정) | final(확정)

training_cases (학습 사례)
  └─ training_revisions (단계별 본문)

calibration_data (합격 자소서 — 평가 레퍼런스용)
style_profile      (학습 데이터에서 추출한 규칙 JSON)
```

학습 사례는 컨설팅 별 자동 등록이 아니라 **확정 후 명시적 토글로만 등록**.

## 4. 첨삭 워크플로우 (사용자 관점)

```
[입력]
회사·직무·공고 입력 → 자소서 붙여넣기/파일 업로드
        ↓
[자동 저장 시작]
'첨삭 시작' 클릭 시점에 Consulting(draft) 자동 생성, draft revision 저장
        ↓
[AI 첨삭]
performRevision 호출 → revisedContent + comments + evaluation 반환
                    → first revision 저장 (재첨삭마다 새 row)
        ↓
[검토/수정]
컨설턴트가 결과 검토. 필요 시 수정 (1.5초 디바운스로 second revision 자동 저장)
또는 'AI 재첨삭' (수정본을 다시 AI에 → 같은 프롬프트로 single-pass 재실행)
        ↓
[확정]
'이대로 확정' 또는 '수정본으로 확정' → final revision 저장 + status=completed
원본 대비 재평가 호출 (Before/After 비교)
        ↓
[학습 등록 (옵션)]
체크박스: "이 사례를 학습 데이터로 추가"
  - 컨설턴트 수정 있으면 기본 ON
  - AI 결과만 확정한 경우 기본 OFF (AI→AI 순환 비추천)
'마무리' 클릭 → 토글 ON 시: TrainingCase + TrainingRevision 생성, 패턴 분석 비동기
```

## 5. 첨삭 AI 호출 구조 (`performRevision`)

### System prompt 구성 (매 호출마다 풀로 전송)

```
1. 컨설턴트 역할 정의 ("한 글자·한 단어의 어조까지 정밀 첨삭")
2. style_profile 주입 (DB JSON, 5분 캐시)
   ├─ 어미 변환 규칙 (예: "~할 수 있었습니다" → "~했습니다", 14/18건)
   ├─ 단어 교체 사전 (예: "적극적으로" → "주도적으로", 8/18건)
   ├─ 삭제/추가 권장 표현
   ├─ 구조 규칙 (소제목 형식 등)
   ├─ 직군별 어조
   └─ 문장 패턴
3. 하드코드 매크로 규칙 (코드 ~25줄)
   ├─ JD를 뼈대로 이식
   ├─ 소제목 [행동/기술 + 정량 성과] 형식
   ├─ 도입부/마무리 패턴
   ├─ 에피소드 1개 집중
   ├─ 기업 고유 자산 발굴
   └─ 문제-원인-해결 구조
4. 하드코드 미시 규칙 (코드 ~40줄)
   ├─ 종결 어미 공식
   ├─ 단어 교체 (감성→비즈니스, 구어→격식)
   ├─ 모호→수치 (Before→After)
   ├─ 수식어 삭제/추가
   ├─ 조사·접속사 정리
   ├─ 주어 빈도 축소
   └─ 문장 길이 조정
5. 직군별 어조 (IT/금융 RM/금융 PB/공공/마케팅/CS)
6. 절대 금지: 내용 날조 (원문에 없는 경험 창작 X, 추론 수치화 OK)
7. 과거 컨설팅 사례 3건 (buildTrainingExamples)
   ├─ 고객별 우선 → 없으면 전체 최신 3건
   └─ 각 사례의 draft/first/second/final 전체 내용 포함 (사례당 수천 토큰)
8. JSON 응답 스키마 + evaluation 작성 규칙
```

### User prompt

```
REVISION_INSTRUCTION (체크리스트 7섹션 — 구조개편/표현/단어/조사/주어/문장길이/보완요청)
+ 회사·직무·JD·컨설턴트 메모
+ 자소서 원문
+ "JSON 형식으로만 응답"
```

### 모델·파라미터

- 모델: `claude-sonnet-4-20250514` (max_tokens 8192)
- 응답: JSON `{revisedContent, comments[], evaluation{}}`

---

## 6. 평가 호출 (`performEvaluation`)

확정 후 한 번 더 호출됨. 5축 루브릭(구체성/논리성/직무연관성/차별성/표현력) × 1-5점.

System prompt에 합격 자소서 + 학습 final을 레퍼런스로 주입:
- 같은 회사 합격 자소서 우선
- 부족하면 같은 산업군에서 보충
- 학습 final도 최신 2건 추가

원본이 같이 들어가면 "원본 대비 개선/퇴보" 비교 평가 추가.

---

## 7. 패턴 분석 (`analyzeTrainingPattern`)

마무리 시(학습 토글 ON) 비동기 호출. 원본+최종을 짧게 비교해 3줄 요약(구조 변화 / 어조 변화 / 핵심 패턴).

**현재 한계**: 결과를 토스트로만 표시. `style_profile`은 자동 갱신되지 않음 (수동 스크립트 `build-style-profile.ts` 실행 필요).

## 8. 최근 메커니즘 변경 (2026-05-01)

### Before
- 확정 → 명시적 "저장" 클릭 시점에만 Consulting/Revision/TrainingCase 일괄 생성
- 그전까진 메모리만 (앱 닫으면 작업 증발)
- 저장 = 학습 등록 (자동, 무조건)

### After
- 첨삭 시작 시점에 Consulting 자동 생성 (draft 상태)
- 모든 단계(draft/first/second/final) 자동 저장
- 저장과 학습 등록 분리:
  - 저장: 자동
  - 학습 등록: 확정 후 명시적 체크박스 + '마무리' 버튼

### 부수 변경
- 인라인 고객 등록 시 회사·직무 차 있으면 Consulting도 같이 생성 (이력 즉시 반영)
- 재첨삭 시 `originalContent` 덮어쓰기 버그 수정 (`aiInputContent` 별도 추적)
- 새 첨삭 시작 시 `client.targetPosition` 자동 채움
- 백엔드: `PUT /consultings/:id`, `PUT /revisions/:id` 신설

## 9. 의논 중인 고도화 로드맵

### Phase 1: 인프라 업그레이드
- **모델 분리**: 첨삭 = Opus 4.7 + extended thinking / 평가 = Sonnet 4.6 / 패턴 = Haiku 4.5
- **프롬프트 캐싱**: 시스템 프롬프트가 거의 동일하므로 큰 효과 기대
- **학습 사례 선택 로직**: 회사·직무·산업 매칭 가산점

### Phase 2: 전문가 노하우 레이어 (★ 핵심)
현재 시스템이 가진 지식:
- 하드코드 룰: "WHAT to change" (표면 변환)
- style_profile: "통계적으로 자주 일어나는 변환" (빈도)

빠진 것: **"WHEN/WHY/HOW to decide"** — 판단 프레임워크.

→ 사용자(컨설턴트 본인)와의 인터뷰로 추출하여 시스템 프롬프트의 *최상위 권위 레이어*로 박을 예정.
→ 인터뷰 질문은 별도 문서 `EXPERT_INTERVIEW.md` 참조.

### Phase 3: 학습 루프 폐쇄
- `analyzeTrainingPattern` 결과를 `style_profile`에 자동 누적
- 마무리 시점 또는 N건 누적 시 트리거

### (참고) 폐기된 Phase 4: 재첨삭 차별화
1차/재첨삭/최종 단계별 다른 프롬프트 → 폐기.
이유: 단계 분할은 인간 컨설턴트의 인지 부하 분산을 위한 작업 흐름 흔적이지, AI에 필요한 게 아님. AI는 한 번에 완벽한 결과를 뽑는 게 목표.
대신 **재첨삭 시 컨텍스트 보강**: "이전 AI 첨삭본 + 컨설턴트 수정 부분 diff"를 프롬프트에 추가하여 single-pass 품질 향상.

## 10. 시스템 프롬프트 권위 우선순위 (제안)

세 소스가 동시 존재할 때 충돌 처리:

```
1순위 (최상위): 전문가 노하우 (사용자 직접 작성한 판단 프레임워크)
2순위:        style_profile (학습 데이터 추출 규칙)
3순위:        하드코드 룰 (코드의 매크로/미시 규칙)
```

배치 순서: 시스템 프롬프트의 가장 *마지막*에 노하우를 두어 recency bias 활용.

## 12. 토론 시 던질 만한 질문 (예시)

- 시스템 프롬프트가 매우 길다(수천 토큰). 모델이 후반부 룰을 무시할 위험은? 캐싱과 별개로, 룰을 줄이거나 압축할 수 있나?
- "한 번에 완벽한 결과"라는 목표 vs "사용자에게 안 보이는 internal staging"(extended thinking) — 후자가 정말 품질 향상인가?
- 학습 데이터가 18쌍밖에 없는데 in-context few-shot으로 충분한가? RAG/임베딩 기반 검색을 도입할 가치가 있는가?
- 컨설턴트의 "암묵지"를 인터뷰로 추출하는 게 정말 작동할까? 다른 방법(녹화 + 분석, 작업 중 내레이션 요청)이 더 정확하지 않을까?
- AI 첨삭 결과의 *품질을 측정*하는 자체 지표가 없다. 평가 점수 외에 "몇 % 컨설턴트가 손대지 않고 통과시켰는가" 같은 지표를 도입할 가치가 있나?
